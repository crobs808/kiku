# Kiku v0.1.11 - Incremental Android Update (Alpha Pre-Release)

Release date: 2026-04-27

## Summary
This incremental release focuses on Android background resiliency and download stability. It adds a foreground service notification path for long-running app activity and hardens model downloads with retry + resume behavior.

## Scope
- Platform: Android
- Release type: Incremental update on top of `v0.1.10`
- Focus areas:
  - Background execution resiliency (listening/transcribing + model downloads)
  - Model download reliability under flaky network/background transitions

## Key Changes

### Added: Foreground Service Notification Keepalive
- Introduced a dedicated Android foreground service (`KikuForegroundService`) with a persistent low-priority notification while critical work is active.
- Added Android foreground service declarations and permissions for:
  - `dataSync`
  - `microphone`
  - `mediaProjection`
- Added runtime bridging so Rust/Tauri can update foreground-service activity types based on current app work.
- Expanded keepalive logic to cover not only model download, but also active listening/transcribing flows.

### Added: Download Retry + Resume Resiliency
- Model downloads now support automatic retry with bounded backoff.
- Model downloads now resume from partial bytes (`.part`) using HTTP Range requests when supported.
- Transient network/read failures no longer fail immediately; they retry first and preserve partial progress.

### Changed: Unified Activity Keepalive Coordination
- Replaced download-only keepalive behavior with centralized activity-state reconciliation.
- Keepalive state now reacts to session transitions and source toggles (mic/system audio) so service types stay accurate.
- Listening path is pre-armed before start to reduce race conditions when transitioning into background-sensitive capture states.

## User-Visible Behavior
- While downloading a model, you should now see a foreground notification indicating background activity.
- While listening/transcribing (including system audio capture), a foreground notification should remain active.
- Switching apps should be significantly less likely to interrupt model download or active transcription flows.
- If network hiccups occur during model download, the app should retry and continue from partial progress instead of failing near the beginning.

## Verification Notes
Recommended smoke checks for this incremental release:
1. Start model download, background the app, and confirm download progress continues.
2. Start listening with microphone enabled, background the app, and confirm session remains active.
3. Enable system audio capture, start listening, background the app, and confirm no immediate stop.
4. During model download, briefly disrupt network connectivity and confirm retry/resume behavior.

## Known Limitations
- This update improves resilience but does not guarantee survival after explicit force-stop or severe OS process reclamation scenarios.
- Android/Gradle Java 8 source-target deprecation warnings remain unchanged from prior release.
